# 既有项目接管规范

目标文件夹非空、存在 Git 历史或已有 Codex 项目任务时，使用本规范。

## 全项目文件覆盖

把“读取整个项目”解释为可验证覆盖，绝不能鲁莽占满上下文。

1. 用 `scripts/project_inventory.py` 清点全部项目自有路径。
2. 记录被排除的依赖、生成物、缓存和版本控制目录。
3. 疑似秘密或凭据只登记元数据；除非当前任务得到用户明确授权，否则不读取内容。
4. 优先读取 `AGENTS.md`、README、架构文档、项目清单、构建测试配置和入口文件。
5. 检查 Git 状态、当前差异和近期提交，理解正在进行的工作。
6. 中小型项目分批读取所有相关文本源码、配置、测试和文档。
7. 大型项目按子系统划分，先读入口、当前工作和高风险路径，再把独立只读覆盖派给员工。
8. 维护覆盖账本：完整读取、抽样、仅元数据、已排除、无法读取、仍未知。

不得为了熟悉项目打开 `.env`、私钥、凭据存储、访问令牌、Cookie 或其他秘密配置。不得把 vendor 或生成目录当作作者项目知识载入。

## 推断总指挥类型

根据文件、文档、任务历史和当前交付物推断总指挥职责。

| 项目证据 | 总指挥类型 | 常见员工岗位 |
| --- | --- | --- |
| 应用、服务、网站、游戏、软件包 | 产品研发总指挥 | 产品、架构、开发、测试、发布、安全 |
| 活动、帖子、脚本、内容日历 | 内容运营总指挥 | 调研、选题、写作、视觉、审核、分发 |
| 数据集、Notebook、SQL、看板 | 数据分析总指挥 | 数据质量、分析、可视化、报告、验证 |
| 剧本、分镜、素材、音频 | 影视制作总指挥 | 编剧、导演、摄影、声音、剪辑、连续性 |
| `SKILL.md`、插件清单、MCP、智能体工具 | Codex Skill 总指挥 | 调研、流程架构、编写、校验、文档、发布 |
| 制度、排班、审批、业务记录 | 业务运营总指挥 | 流程、数据、实施、审计、测试、上线 |

混合项目使用混合岗位。按独立交付物选择三至六名员工，不使用固定模板。

所有项目类型都必须增加 `员工00｜Token监管与模型路由｜项目名`。这是必备只读控制岗位，不替代负责实际交付物的员工。

归整任务前形成：

```text
总指挥职责画像
项目：
类型：
生命周期阶段：
主要目标：
当前工作：
技术与工具：
关键风险：
验收表面：
```

## 归整现有任务窗口

按准确项目路径过滤后再分类。

### 指挥部

重命名当前任务后，结合项目路径和活跃状态识别当前 thread。只置顶已确认的当前任务，绝不自动接管另一个总指挥。

### 结构化员工

复用标题为 `员工NN｜职责｜项目名` 且近期历史仍匹配岗位的任务。项目范围变化时刷新岗位配置。

### 新建或未结构化候选

只有同时满足以下条件才接纳：

- 属于准确项目路径；
- 当前活跃或最近更新；
- 近期历史清楚匹配需要的职责；
- 改作员工不会破坏一个独立历史成果。

先发送岗位配置，再重命名。用途不明时保留原名并报告。

### 历史任务

保留已完成、以成果为中心的历史任务。它们是项目记忆，不是自动员工。

### 禁止的清理操作

自动归整期间不得归档、删除、合并、替换或接管任务。这些操作必须由用户另行明确授权。

## 首次启动顺序

1. 匹配本地项目。
2. 清点文件和 Git 状态。
3. 读取项目规则、说明、清单和当前工作。
4. 推断总指挥职责画像。
5. 重命名、识别并置顶指挥部。
6. 清点并阅读相关任务窗口。
7. 在其他昂贵派工前，先复用或创建唯一的 Token 监管员工。
8. 让 Token 监管员识别重复岗位，并为其余岗位建议最低足够模型档位；由总指挥执行决定。
9. 复用、接纳或创建最小有效的实际交付员工编制。
10. 员工登记结束后稳定标题。
11. 通过岗位配置消息分配模型、推理、Token 预算与止损基线。
12. 验证标题、项目路径、岗位确认和置顶结果。
13. 汇报职责画像、覆盖账本、员工名单、模型配置、Token 控制与排除项。

## 后续启动顺序

1. 刷新文件清单、Git 状态和近期变化。
2. 重新评估生命周期和职责画像。
3. 发现新建和未结构化项目任务。
4. 不归档、不替换历史地完成归整。
5. 让 Token 监管员审计上一轮后的重复读取、重复任务、重试和模型过度使用。
6. 职责变化时刷新员工岗位与模型基线。
7. 只补缺失岗位，避免重复员工。
8. 再次置顶指挥部并核对标题。
9. 汇报本轮变化以及避免了哪些 Token 浪费。
