# 现代化三省六部治理架构

本架构借用“三省六部”的职责分离思想，为复杂或高风险 Codex 项目增加“立案—审议—执行—验收”治理链。它是现代项目治理类比，不是历史制度复刻，也不是要求机械创建九个任务窗口。

## 启用与持久化

用户可以在启动命令中加入“`三省六部架构`”或“`三省六部模式`”，例如：

```text
我的总指挥，效率模式，三省六部架构
```

当前任务已经是活跃总指挥时，用户只发送“`三省六部架构`”或“`三省六部模式`”，应校准现有组织，绝不能创建第二名总指挥或重复员工。

把 [治理模板](../assets/GOVERNANCE.template.md) 复制或校准为：

```text
.codex/project-commander/GOVERNANCE.md
```

只有总指挥可以写入。默认保持本地且不纳入 Git，不保存秘密、原始对话全文或隐藏思维过程。

## 三省：决策与执行分离

| 三省职能 | 现代职责 | 必须交付 | 权限边界 |
| --- | --- | --- | --- |
| 中书省｜立案与方案 | 把用户目标转换成完成定义、依赖图、任务合同、风险与候选方案 | 可审议方案包 | 不批准自己的高风险方案，不直接扩张执行范围 |
| 门下省｜审议与封驳 | 独立检查范围、证据、冲突、Token 预算、权限和验收标准 | `准行`、`封驳`或`补证`决定及理由 | 默认只读；不得把封驳变成自行改写用户目标 |
| 尚书省｜执行统筹 | 只对获准工作拆分、排期、派工、中转交接、集成和状态核对 | 部级任务包、派工记录、集成证据 | 不得绕过审议门或擅自改变已批准目标 |

总指挥是三省之间唯一中转和最终责任人。用户始终是最高授权来源；历史称谓不产生额外权限。

## 六部：项目职能池

| 六部职能 | 现代职责 | Codex 中的典型成果 |
| --- | --- | --- |
| 吏部｜组织与任用 | 花名册、岗位独立性、容量、标题、接纳与改派建议 | `ORG_CHART.md`、岗位合同、缺口/重叠报告 |
| 户部｜Token 与资源 | Token 监管、模型路由、上下文复用、预算、数据与资产清单 | 唯一员工00的预算审查、降档/止损建议 |
| 礼部｜标准与沟通 | 文档、体验、内容规范、统一汇报、对外交付包装 | 文档、设计/内容标准、汇报稿；外发仍需用户授权 |
| 兵部｜生产与执行 | 代码、内容、分析、媒体制作或其他主要生产工作 | 有边界的可验证生产产物；每块写范围只有一名负责人 |
| 刑部｜质量与风险 | 独立 QA、安全、合规、缺陷分级、证据门 | 只读审查、测试证据、风险裁决；修复必须另行派工 |
| 工部｜工程与基础设施 | 工具、构建、自动化、集成、环境与发布准备 | 构建脚本、自动化、集成产物；部署/发布仍需用户授权 |

员工00始终只有一名，并映射到户部职能；不得再创建第二个 Token 监管窗口。六部是职能标签，不是六个强制部门经理。

## 治理门与任务生命周期

1. **立案**：中书省职能形成方案包，包含目标、完成定义、依赖、冲突、唯一负责人候选、预算和验收。
2. **审议**：门下省职能独立输出 `准行`、`封驳`或`补证`。高风险任务的方案作者不能兼任最终审议者。
3. **裁决**：总指挥处理封驳或补证；只有 `准行` 项才能进入执行队列。
4. **执行**：尚书省职能把已批准工作映射到一个部、一个部门和一名生产负责人，并建立书面交接包。
5. **流转**：任意员工完成后，总指挥立即验收并解锁后续就绪任务，不等待无关员工。
6. **质量门**：刑部或门下省职能依据证据复核。生产者不得独自批准自己的高风险交付物。
7. **结案**：总指挥核对任务账本、治理记录和真实验收表面后统一汇报。

任务账本中的每一行都必须记录治理门状态。`封驳`或`补证`不能派工；发生范围、预算或风险实质变化时必须重新审议。

## 按项目规模折叠，而不是凑窗口

| 场景 | 建议映射 | 约束 |
| --- | --- | --- |
| 小型/节省模式 | 2–3 个交付窗口兼任三省六部职能 | 高风险方案与独立验收不能由同一窗口承担 |
| 中等/普通模式 | 3–5 个交付窗口，分开方案、生产和质量 | 员工00不计入交付 WIP |
| 效率模式或复杂项目 | 5–7 个有真实任务的交付窗口 | 不机械创建九个窗口；无工作职能标为未设岗 |

一个员工可以兼任相容职能，但仍只能有一个所属部门、一个主要负责成果、一块可写范围和一项当前主任务。不相容组合包括：高风险方案作者与最终审议者、生产者与自己产物的独立质量裁决者、Token 监管员与生产负责人。

## 来自公开 GitHub Skills 的可复用机制

本架构吸收以下公开项目中的通用做法，而不依赖其特定命令或平台：

- [CCPM](https://github.com/automazeio/ccpm)：显式依赖、并行性、冲突、就绪/阻塞状态和完成后解锁。
- [TDD Multi-Agent Orchestration](https://github.com/glebis/claude-skills/blob/main/tdd/SKILL.md)：上下文隔离、阶段分离、人工检查点和失败止损。
- [Code Review and Quality](https://github.com/addyosmani/agent-skills/blob/main/skills/code-review-and-quality/SKILL.md)：独立审查、证据和多维质量门。
- [Quality Playbook](https://github.com/github/awesome-copilot/blob/main/skills/quality-playbook/SKILL.md)：角色图、阶段入口/出口条件、书面交接和持久进度记录。

这些来源提供设计启发；本技能仍以 Codex 独立侧边栏任务窗口、总指挥唯一写入账本和用户授权边界为准。
