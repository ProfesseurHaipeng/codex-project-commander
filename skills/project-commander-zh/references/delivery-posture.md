# 交付策略与上线优先协议

本协议用于包含部署、发布、上线或生产变更的任务。它独立于运行模式：运行模式控制 WIP、模型与 Token；交付策略控制检查深度、上线顺序和故障处置。

## 选择策略

| 策略 | 目标 | 默认检查范围 |
| --- | --- | --- |
| 上线优先 | 尽快获得可用线上结果 | 最小上线门槛，通过后立即部署，非阻断项转入上线后补强 |
| 平衡交付 | 兼顾速度和可靠性 | 变更路径检查、构建、关键冒烟，通过后部署 |
| 严格发布 | 先降低高影响风险 | 更广泛的测试、安全、合规、性能或迁移验证后部署 |

用户明确说“先上线”“赶紧部署”“上线优先”时直接采用上线优先，不重复询问。用户明确要求完整验证、合规审查或严格发布时采用严格发布。若时间优先级会实质改变方案但用户没有表达清楚，只问一次：

> 这次以尽快上线为主，还是先完成更完整检查？

未回答但存在安全、可逆的路径时，说明假设并采用平衡交付。交付策略词本身只改变方案，不扩大权限；部署、发布或生产变更仍需用户明确授权。

## 上线优先的最小门槛

只检查与本次变更相关且适用的项目，不执行无关的全套审计：

- 交付物能够构建或启动，关键用户路径完成一次冒烟；
- 没有已知泄露的密钥或凭据；
- 涉及身份、权限时完成最小授权检查；
- 涉及资金、隐私、数据丢失或法律合规时，相关关键风险不得延期；
- 破坏性数据库或 Schema 变更具有已验证的备份、恢复或等价恢复路径；
- 存在实际可用的修复前进方案，必要时具备回滚能力；
- 用户已经明确授权本次部署或上线。

任何适用底线失败时只报告具体阻断项，不借机扩大成全项目安全审计。解决阻断后只重跑受影响检查。

## 可延期工作

以下项目通常可以在上线后处理：与变更路径无关的全量测试、非关键重构、低严重度警告、可选文档、视觉边角问题、非阻断性能优化，以及不影响本次上线的广泛加固扫描。

在任务账本的“上线后补强”中记录：事项、严重度、延期原因、负责人、证据、触发条件或期限、下一动作。合并相关低风险事项，不为每条小问题单独创建员工或立即派工。

## 修复前进与回滚

对可逆、非关键缺陷优先修复前进，不因小问题反复回滚和重新部署。只有出现持续性重大伤害时才回滚，例如完全不可用、数据损坏或丢失、身份/支付/隐私/关键安全事故、法律风险，或用户明确要求。

同一路线两次实质相同的部署、回滚或检查循环后停止该路线，保留证据并重规划。不得用反复回滚制造“严谨”的表象。

## Token 约束

- 复用仍然有效的最近构建、测试和发布证据；未变化的全套检查不重复运行。
- 每个适用上线门槛先执行一次最小定向检查；修复后只重跑受影响部分。
- 常规打包、清单和重复发布准备优先 Luna；有边界的发布和数据工作优先 Terra；复杂事故、高风险迁移或关键裁决才使用 Sol。
- 把上线后补强压缩成批次，不让低风险尾项长期占用高档模型或多个窗口。

## 上线后闭环

上线后先验证真实线上健康与关键路径，然后更新账本：记录版本、时间、证据、已知问题、补强队列和负责人。若队列只剩可延期事项，向用户汇报已上线、当前风险与后续安排，不把“所有优化全部完成”伪装成上线前置条件。
