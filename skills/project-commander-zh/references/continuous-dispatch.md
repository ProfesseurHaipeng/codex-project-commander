# 持续派工与持久任务账本

总指挥收到、恢复或重新规划多步骤任务时，使用本协议。用事件驱动派工代替整批等待，同时保证每名员工一次只执行一条主任务线。

## 建立任务账本

把 [任务账本模板](../assets/TASK_LEDGER.template.md) 复制到：

```text
.codex/project-commander/TASK_LEDGER.md
```

需要时创建父目录。只有总指挥可以写账本。员工通过各自任务窗口汇报，由总指挥验证证据后更新状态。

账本默认保持本地、不纳入 Git。未经用户授权不得修改 `.gitignore`。提交账本前提醒用户，其中可能含项目计划或内部证据位置。不得写入秘密、凭据、隐藏思维过程、原始对话全文或文件正文。

出现以下事件时更新账本：

- 总指挥接受或修改用户目标；
- 子任务进入待派、工作中、受阻、停滞、待验收或完成；
- 员工被分配、释放或改派；
- 依赖、模型档位、推理强度、重试上限或验收方式变化；
- 运行模式变化。
- 部署授权、交付策略、最小上线门槛或上线后补强变化。

## 表达完整任务

把用户目标拆成依赖图。每项子任务必须包含：

- 稳定任务 ID；
- 一个可验收结果；
- 依赖任务 ID；
- `.codex/project-commander/ORG_CHART.md` 中的部门与主岗位；
- 唯一负责人或“未分配”；
- 一个状态：`待规划`、`待派`、`工作中`、`受阻`、`待验收`、`完成`、`取消`或`停滞`；
- 一个治理门状态：标准架构填 `不适用`；三省六部架构填 `待审`、`准行`、`封驳`或`补证`；
- 拥有文件或职责范围；
- 模型档位、推理强度和预算等级；
- 下一个有意义的检查点；
- 验收证据与下一动作。

每名员工一次只接收一项正在执行的主任务。后续任务先放入账本队列，不要同时向一个窗口塞入多份任务合同。相关后续工作优先续派给已熟悉该子系统的员工，但不得造成写入冲突。

只有全部依赖均已完成、所需权限和输入已经具备，且适用的治理门为“准行”时，任务才进入“待派”。“封驳”或“补证”任务不得派出。真实集成屏障必须等待；不得人为制造“所有员工一起结束”的整批屏障。

## 选择运行模式

默认使用中等模式。“普通模式”是中等模式的别名。模式只控制并行在制任务数、续派速度、检查频率和 Token 策略，绝不扩大权限、降低安全、跳过验收或绕过止损。

| 模式 | 中文触发词 | 实际交付任务 WIP | 模型与 Token 策略 | 检查点 |
| --- | --- | --- | --- | --- |
| 节省模式 | `节省模式` | 1–2 项 | Luna 优先；有依据才用 Terra；默认 Light/低 | 稀疏、按里程碑 |
| 中等模式 | `中等模式`、`普通模式` | 2–3 项 | 按任务选择 Luna/Terra；确有需要才用中推理 | 正常的有意义检查点 |
| 效率模式 | `效率模式` | 3–5 项互不冲突任务 | 完成即续派；有边界工作默认 Terra；有依据才用 Sol | 更频繁读取事件/检查点，但不空转轮询 |

Token 监管员工不计入实际交付任务 WIP。文件所有权、依赖、工具限制或风险要求更少并发时，主动降低 WIP。不得为了填满额度而制造无价值任务。

允许把模式与启动命令组合，例如“我的总指挥，效率模式”。总指挥已经启动时，单独发送模式触发词只修改账本与后续路由，不创建第二名总指挥或重复员工。

交付策略是独立维度。部署类任务按 [交付策略协议](delivery-posture.md) 记录“上线优先、平衡交付或严格发布”；单独切换策略不改变 WIP，也不构成部署授权。

## 运行持续派工循环

完成任务拆解并写入账本后：

1. 重新计算所有“待派”任务。
2. 按依赖解锁价值、用户价值、风险降低、上下文复用和最低足够模型排序。
3. 用部门、岗位与可用员工匹配且互不冲突的任务填充当前模式允许的 WIP。
4. 每名员工只发送一份任务合同，并把状态改为“工作中”。
5. 启用 [完成监听器](completion-watchdog.md)，由总指挥心跳发现完成或有意义检查点；不得等待员工跨窗口主动推送。
6. 任意员工完成后，立即验证其报告和证据。
7. 验收通过就标为“完成”、释放其拥有范围并重新计算依赖。
8. 立即把价值最高且岗位适配的“待派”任务续派给该员工或另一名合适的空闲员工。
9. 按 [空闲容量审计](completion-watchdog.md#空闲容量审计) 检查其他看似空闲员工；有真实适配任务就补派，没有就标记休息。
10. 验收失败时改为“待验收”“受阻”或“停滞”，记录证据并在重试上限内重新规划。
11. 持续循环，直到所有必需任务已完成、经授权取消，或明确受阻于用户/外部状态。

不能等待最慢的员工才继续无关工作。只有某个下游任务明确依赖当前全部任务时，才等待它们全部结束。

## 巡检进度但不空转轮询

不得主观给员工贴“偷懒”标签，只按可观察证据分类任务状态。

为每名员工记录：

- 当前任务和开始事件；
- 最后一次有意义更新或证据；
- 承诺的下一个检查点；
- 阻塞、重试次数与新假设；
- 下一项适配的待派任务。

监听器发现员工最新轮次结束、到达承诺检查点、需要依赖决定，或与最新项目状态矛盾时才读取进度。使用任务 ID 和最后已处理报告标识去重。定时心跳只负责发现变化，禁止没有新证据的定时催问。

检查点已过却没有实质进展时：

1. 发送一次简短询问，要求当前证据、阻塞和下一动作；
2. 下一个有意义检查仍无新证据时，标记为“停滞”；
3. 保留有效证据，按条件触发止损，然后缩小、重规划、降档、有依据地升档或改派；
4. 记录为什么变化预计能够改善结果。

只读待命员工是“待命”，不是“停滞”。等待明确依赖的任务是“受阻”，不是“停滞”。

## 恢复与校准

启动或恢复总指挥时：

1. 存在账本就先读取；
2. 把每个未结束条目与当前文件、Git 状态及对应任务窗口核对；
3. 把条目的部门和负责岗位与当前组织架构核对；
4. 保留稳定任务 ID 和证据位置；
5. 修正过期部门、负责人、状态、依赖和模型分配；
6. 加入新发现的用户要求与项目任务；
7. 重新计算待派队列并恢复持续派工循环。
8. 检查唯一完成监听器；补收停用期间未处理的员工报告，并按当前模式恢复或暂停监听。

账本是简洁运营索引，不是绝对事实。旧条目与当前项目证据或已验收员工结果冲突时，以后者为准。
