# Codex 更便利的使用方式

Codex 真正高效的用法，不是把所有需求塞进一个长对话，而是让“项目、任务、长期规则、重复工作流和验证证据”各自待在正确的位置。

## 1. 一个项目承载共同文件，一个任务只负责一个结果

当工作会持续多天、包含多个输出，或多个任务需要访问同一文件夹时，建立本地项目。项目负责共享文件上下文，任务负责保存各自的对话和执行历史。

推荐拆法：

- `产品需求与范围确认`
- `技术架构与数据模型`
- `功能开发`
- `测试与发布检查`

不要把这些内容全部堆进一个无限增长的任务窗口。任务越聚焦，Codex 越容易记住当前目标、边界与完成条件。

官方参考：[Projects, chats, and tasks](https://learn.chatgpt.com/docs/projects)

## 2. 提示词先写“结果”，再写过程

一个实用任务至少应包含四件事：

```text
目标：最终要得到什么？
范围：允许修改哪些文件、系统或页面？
完成标准：什么证据能证明完成？
限制：哪些操作不能做？
```

例如：

```text
修复结算页总价与 API 不一致的问题。
只允许修改 client/checkout 与 server/pricing。
必须补回归测试，并在浏览器中验证最终金额一致。
不要修改数据库、部署生产环境或改动支付接口。
```

这比“帮我看看结算页”更容易得到可验收结果。

## 3. 把长期规则写进 AGENTS.md

如果同一个要求需要反复提醒，就不应继续依赖聊天记忆。把它写进 `AGENTS.md`：

- 全局个人规则：`~/.codex/AGENTS.md`
- 项目共享规则：项目根目录的 `AGENTS.md`
- 子模块特殊规则：更靠近对应目录的 `AGENTS.md`

适合写入的内容包括：

- 项目目录结构；
- 安装、构建、测试和 lint 命令；
- 技术与代码风格约定；
- 禁止事项；
- 完成定义与验证方法；
- 发布与评审流程。

保持规则短、准确、可执行。Codex 犯同类错误两次时，应做一次复盘并更新规则，而不是继续重复口头提醒。

官方参考：[Custom instructions with AGENTS.md](https://learn.chatgpt.com/docs/agent-configuration/agents-md)

## 4. 把重复工作流做成 Skill

`AGENTS.md` 适合“这个项目一直要遵守什么”，Skill 适合“遇到某类任务时应该执行哪套流程”。

适合做成 Skill 的例子：

- 每周经营数据复盘；
- PR 评审与修复；
- PDF、表格、演示文稿制作；
- 项目总指挥与多任务窗口调度；
- 固定品牌视觉或内容生产流程。

最可靠的调用方式是显式写出技能：

```text
使用 $project-commander-zh 启动当前项目的总指挥。
```

自然语言也可能触发技能，但 Codex 会限制初始技能列表占用的上下文；安装技能很多时，一些描述可能被缩短或省略。因此关键工作流不要只依赖隐式匹配。

官方参考：[Build skills](https://learn.chatgpt.com/docs/build-skills)

## 5. 为任务命名，而不是依赖自动标题

任务名称应描述结果或岗位，例如：

- `总指挥｜提醒我`
- `员工03｜架构与方案｜提醒我`
- `结算页金额一致性修复`

任务的首次回复可能触发自动标题，所以程序化创建任务时，应等待首轮结束后再命名，并读取任务列表复查最终标题。

固定返回的任务可以置顶；已经完成且不再使用的任务可以归档。置顶只改变侧边栏位置，不会增加任务上下文。

## 6. 模型与推理强度按任务分配

更强模型和更高推理强度并不适合所有工作。建议使用“最低足够能力”原则。GPT-5.6 家族可用时，按 Sol、Terra、Luna 三级路由：

| 工作类型 | 优先模型 | 推理强度 |
| --- | --- | --- |
| 文件清单、提取、格式转换、批量跑量 | Luna | Light/低 |
| 数据整理、常规实现、文档、测试、调研 | Terra | 中 |
| 复杂软件、跨系统调试、迁移、集成 | Sol | 中；有证据才升高 |
| 安全、高风险最终判断 | Sol | 有依据才用高或 Extra High |

出现以下情况再升级：

- 目标有关键歧义；
- 验证连续失败；
- 工作跨越多个系统；
- 涉及安全、隐私、资金、法律或生产数据；
- 多份证据相互冲突。

每次升档都应写明低一档为什么不能通过验收。不要自动使用 Max；不要让员工使用可能产生第二层委派的 Ultra。当前环境没有 GPT-5.6 时，把三级策略映射到可用的高、中、低成本等价模型，不能虚构模型 ID。

官方参考：[Models](https://learn.chatgpt.com/docs/models)

## 7. 并行适合只读探索，写入必须划清所有权

适合并行：

- 代码库探索；
- 测试缺口检查；
- 日志分析；
- 竞品与资料调研；
- 多个互不依赖的文档审阅。

不适合无边界并行：

- 多个任务同时修改同一文件；
- 多个任务同时重构同一模块；
- 没有统一集成者的跨模块修改。

并行写入时，每个文件或子系统必须只有一个明确负责人。其他窗口可以只读评审，最终由一个集成负责人合并和验证。

## 8. Local 与 Worktree 的选择

Local 任务直接使用当前工作目录，适合共享当前未提交状态，但并行写入容易冲突。

Worktree 提供隔离的 Git 工作区，适合彼此独立的开发任务。使用前要确认任务能够从目标分支或默认分支开始；不要假设未提交改动会自动出现在新工作树中。

简单原则：

- 共享当前未提交状态：Local，并限制为一个写入者。
- 独立功能、独立文件范围：Worktree。
- 只读调研：Local 或 Worktree 均可，以最少开销为准。

## 9. 要求证据，而不是接受“完成了”

Codex 的完成报告至少应包含：

- 改了什么；
- 修改了哪些文件或外部状态；
- 运行了哪些测试或检查；
- 测试结果；
- 仍然存在的风险；
- 用户如何直接查看结果。

不同任务需要不同证据：

- 代码：diff、测试、构建结果；
- 网站：真实浏览器页面与关键流程；
- 文档：渲染后的版式；
- 视频：实际播放与视听检查；
- 数据：来源、计算过程、质量检查。

导出成功、命令退出码为 0 或员工声称完成，都不能自动等于最终验收通过。

## 10. 用“总指挥”承担协调成本

当一个项目需要多个长期任务窗口时，可以使用本仓库的 Project Commander：

1. 用户只向总指挥给出目标。
2. 总指挥拆分任务并决定是否需要员工窗口。
3. 每个员工收到包含范围和验收标准的任务合同。
4. 员工在自己的任务中工作并提交结构化报告。
5. 总指挥读取结果、验证冲突、要求返工或完成集成。
6. 用户只收到最终结论、证据和真正需要决定的事项。

员工必须是同一项目下的独立任务窗口。独立任务不会自己跨窗口推送消息；总指挥通过任务工具读取和收集结果。

### 先建立组织架构，再创建员工

总指挥应在 `.codex/project-commander/ORG_CHART.md` 中记录组织树、部门、员工花名册、责任与交接图。组织架构决定“谁负责哪类成果”，任务账本决定“当前任务做到哪里”。

推荐采用两层结构：总指挥直接管理治理部门和项目专属交付部门。默认不创建部门经理窗口，以免为管理层本身持续消耗 Token。

每名员工必须具有：

- 一个所属部门；
- 一个主要负责成果；
- 明确输入与输出；
- 独立可写范围与只读依赖；
- 验收责任和模型基线；
- 一次只执行一项主任务的容量。

同一工作项只能有一名生产负责人，可设置只读复核者。跨部门交接由总指挥验收上游结果后，发送压缩交接包给下游员工，不能让员工互相改派。

### 复杂项目可启用“三省六部架构”

输入“我的总指挥，三省六部架构”后，总指挥会建立 `.codex/project-commander/GOVERNANCE.md`。中书省职能负责把目标变成方案，门下省职能独立给出“准行、封驳或补证”，尚书省职能只派发获准任务；六部职能覆盖组织任用、Token与资源、标准沟通、生产执行、质量风险、工程基础设施。

三省六部是职责分离，不是必须创建九个窗口。小项目可以折叠相容职能，但高风险方案作者不能兼任最终审议者，生产者不能独立批准自己的高风险交付物，员工00也不能承担生产工作。

### 用任务账本防止遗忘和整批空等

总指挥应把多步骤任务写入 `.codex/project-commander/TASK_LEDGER.md`，记录稳定任务 ID、依赖、负责人、状态、模型、检查点、验收证据与下一动作。账本默认保持本地、不纳入 Git，并且只有总指挥写入。

每名员工一次只执行一条主任务线。任意员工完成后，总指挥立即验收并续派下一项适配任务；不能因为另一名员工仍在工作，就让已经空闲的窗口等待。只有真实依赖要求一组任务全部完成时才建立屏障。

进度巡检应看“最后实质证据、承诺检查点、阻塞、重试和新假设”，不主观评价员工是否偷懒。错过检查点时先问一次；下一次有意义检查仍没有新证据，才标记为停滞并重规划或改派。

### 三种运行模式

| 模式 | 同时执行的实际交付任务 | 适合场景 |
| --- | --- | --- |
| 节省模式 | 1–2 | Token 紧张、任务清晰、可顺序推进 |
| 中等/普通模式 | 2–3 | 默认日常项目，兼顾速度、质量和成本 |
| 效率模式 | 3–5 个互不冲突任务 | 时间优先、有足够独立工作可持续续派 |

模式决定 WIP、模型倾向和检查频率，不改变权限、安全、文件所有权、验收或止损。效率模式也不能无依据使用 Sol；节省模式也不能跳过必要验证。

### 设置 Token 监管部

总指挥编制中固定保留一名只读 `员工00｜Token监管与模型路由｜项目名`。派工前由它检查：

- 是否已有员工掌握相关上下文；
- 是否重复读取没有变化的文件；
- 是否有多个员工执行相同调查；
- 是否可以用一个任务合并多个重叠任务；
- 是否只需发送增量、文件位置和验收标准；
- Luna 是否足够，不足时 Terra 是否足够；
- 重试何时停止，什么证据才允许升档。

同一方法两次实质相同失败后先止损、保留证据并更换方法，不能继续空转或直接堆高推理强度。只有界面或工具公开真实用量时才报告 Token 数值；否则报告模型、推理强度、轮次、重试、重复读取和重复任务等可观察信号。

### 在长期运行的老项目中接入总指挥

老项目不能直接套用一张固定组织架构。总指挥应先进行项目接管：

1. 清点项目自有文件与目录，读取项目规则、README、架构文档、配置、源码、测试、Git 状态和近期变更。
2. 对依赖目录、生成物、缓存、二进制和敏感文件只做元数据登记，并明确报告未读取内容。
3. 从文件与历史任务判断项目领域、生命周期、当前目标、技术栈、交付物和风险。
4. 形成项目专属的总指挥画像，例如产品研发总指挥、内容运营总指挥、数据分析总指挥或 Skill 开发总指挥。
5. 读取同一项目现有与近期新增任务的摘要，把它们分类为总指挥、结构化员工、新员工候选、历史任务和用途不明任务。
6. 只复用或归整职责明确的任务窗口，不自动归档、删除或接管历史任务。
7. 根据岗位和风险，为员工选择当前 Codex 支持的模型与推理强度基线。
8. 给员工发送岗位配置，再在具体任务中按复杂度动态升级或降级。
9. 自动命名并置顶总指挥，员工保持普通排序，避免整个侧边栏都被置顶占满。

所谓“读取整个项目”应当是可核验的覆盖，而不是把所有字节一次性塞入上下文。小型项目可以分批完整读取全部项目自有文本；大型项目应按子系统建立覆盖表，并明确哪些内容已完整读取、抽样、仅登记元数据、排除或仍未知。

## 11. 常见误区

### 把隐式技能触发当成硬命令

关键流程使用 `$skill-name` 显式调用，或在全局 `AGENTS.md` 中建立明确映射。

### 把子智能体当成长期员工窗口

子智能体适合一次性并行工作。需要在项目侧边栏长期保留、反复派工的岗位，应建立独立任务窗口。

### 老项目一上来就创建一套固定员工

先读项目文件与任务历史，再决定总指挥类型和岗位。否则员工结构会与项目真实阶段错位。

### 一个窗口承担整个项目

项目共享文件，任务聚焦结果。长期项目应按输出和职责拆分任务。

### 多个写入者没有边界

没有文件所有权的并行开发会增加冲突和返工。并行前先切分范围。

### 只看日志，不看最终表面

高价值交付要在用户实际使用的表面验证，例如浏览器、应用、文档查看器或播放器。

## 12. 推荐的最小工作系统

```text
项目文件夹
├── AGENTS.md              # 项目长期规则
├── .codex/config.toml     # 项目配置（需要时）
├── 源代码与资料
└── Codex 项目任务
    ├── 总指挥
    ├── 开发任务
    ├── 测试任务
    └── 调研任务

用户级目录
├── ~/.codex/AGENTS.md     # 个人长期规则
├── ~/.codex/config.toml   # 个人默认配置
└── ~/.agents/skills/      # 用户级 Skills
```

这个系统的目标不是创建更多窗口，而是让每条规则、每个任务和每份证据都有清晰归属。
